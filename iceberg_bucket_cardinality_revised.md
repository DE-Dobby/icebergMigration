# Iceberg 버킷 설계 재검토 — 카디널리티 재조사 결과 반영

> 버킷 컬럼 카디널리티를 재조사한 결과를 반영하여
> 16 / 32 / 64 버킷 설계를 다시 분석한 문서입니다.

## 1. 카디널리티 재조사 결과

| 항목 | 기존 가정 | 재조사 결과 |
|---|---|---|
| 버킷 컬럼 카디널리티 | 5,600 ~ 6,000 | **최대 277** |
| 데이터 87% 구간 | — | **카디널리티 ≤ 10** |
| 데이터 13% 구간 | — | 카디널리티 11 ~ 277 |

> **핵심 변화**: 데이터의 대부분(87%)이 버킷 컬럼의 고유값이 10개 이하인 파티션에 속함

## 2. 카디널리티가 버킷 효율에 미치는 원리

### 실효 버킷 수 개념

Iceberg bucket transform은 `hash(value) % N` 으로 버킷 번호를 결정합니다.
**버킷 컬럼의 고유값 수(카디널리티)가 N보다 작으면, 실제로 생성되는 파일 수는 카디널리티에 수렴합니다.**

```
실효 버킷 수 = min(설정 버킷 수 N, 카디널리티)

예시)
카디널리티=10, bucket(64) → 실효 버킷 수 ≈ 10 → 파일 최대 10개
카디널리티=10, bucket(32) → 실효 버킷 수 ≈ 10 → 파일 최대 10개
카디널리티=10, bucket(16) → 실효 버킷 수 ≈ 10 → 파일 최대 10개

→ 87% 데이터에서는 버킷 수 16 / 32 / 64가 모두 동일한 결과
```

### 그래서 발생하는 문제

```
[87% 데이터 — 카디널리티 ≤ 10]
  bucket(64) 설정해도 → 파티션당 파일 최대 10개 생성
  파티션 데이터가 35GB라면 → 35GB ÷ 10 = 3.5GB/파일 ❌

[13% 데이터 — 카디널리티 최대 277]
  bucket(64) → 파티션당 파일 최대 64개 생성
  파티션 데이터가 35GB라면 → 35GB ÷ 64 = 547MB/파일 ⚠️
```

## 3. 카디널리티 구간별 버킷 수 효과 비교

### 3-1. 저카디널리티 구간 (87% 데이터, 카디널리티 ≤ 10)

| 버킷 수 설정 | 실효 버킷 수 | 파일 수 (예: 35GB 파티션) | 파일 크기 | 판정 |
|---|---|---|---|---|
| **16** | min(16, 10) ≈ **10** | ~10개 | 35GB ÷ 10 = **3.5 GB** | ❌ 과대 |
| **32** | min(32, 10) ≈ **10** | ~10개 | 35GB ÷ 10 = **3.5 GB** | ❌ 과대 |
| **64** | min(64, 10) ≈ **10** | ~10개 | 35GB ÷ 10 = **3.5 GB** | ❌ 과대 |

> **버킷 수를 16에서 64로 늘려도 87% 데이터의 파일 크기는 변하지 않습니다.**
> 이 구간은 **Compaction**으로 별도 관리해야 합니다.

### 3-2. 고카디널리티 구간 (13% 데이터, 카디널리티 11 ~ 277)

| 버킷 수 설정 | 실효 버킷 수 | 파일 수 (예: 35GB 파티션) | 파일 크기 | 판정 |
|---|---|---|---|---|
| **16** | min(16, 277) = **16** | ~16개 | 35GB ÷ 16 = **2.2 GB** | ❌ 과대 |
| **32** | min(32, 277) = **32** | ~32개 | 35GB ÷ 32 = **1.1 GB** | ❌ 과대 |
| **64** | min(64, 277) = **64** | ~64개 | 35GB ÷ 64 = **547 MB** | ⚠️ 허용 |
| **128** | min(128, 277) = **128** | ~128개 | 35GB ÷ 128 = **273 MB** | ✅ 권장 |
| **256** | min(256, 277) = **256** | ~256개 | 35GB ÷ 256 = **137 MB** | ✅ 적정 |

## 4. 16 / 32 / 64 버킷 재분석

### 버킷 수별 그룹별 영향 요약

| 버킷 수 | 저카디널리티 87% (≤10) | 고카디널리티 13% (11~277) | 종합 |
|---|---|---|---|
| **16** | 효과 없음 (실효=10) | 파일 2.2 GB ❌ | ❌ 배제 |
| **32** | 효과 없음 (실효=10) | 파일 1.1 GB ❌ | ❌ 배제 |
| **64** | 효과 없음 (실효=10) | 파일 547 MB ⚠️ | ⚠️ 조건부 채택 |

> **결론**: 16 / 32 / 64 버킷 테스트에서 16과 32는 기존 분석과 동일하게 배제.
> 64는 고카디널리티 파티션에서만 효과가 있으며, 87% 파티션은 Compaction 병행 필수.

## 5. 카디널리티 기반 최적 버킷 수 도출

### 버킷 수 상한선: 최대 카디널리티 = 277

```
버킷 수가 카디널리티(277)를 초과해도 의미 없음
→ 277개 이상의 버킷 설정 = over-bucketing

실용적 상한 = 2의 거듭제곱 중 277에 가장 가까운 값
→ 256 (2^8)
```

### 권장 버킷 수: 64 (현재 테스트 범위 내 최선)

| 고려 요소 | 판단 |
|---|---|
| 87% 저카디널리티 파티션 | 버킷 수 무관 → Compaction 필수 |
| 13% 고카디널리티 파티션 | 64에서 파일 547 MB (허용 수준) |
| Manifest 부담 | 64 기준 낮음 ✅ |
| 테스트 범위(16/32/64) 내 최선 | **64** ✅ |

> 추가 테스트 여건이 된다면 **128 또는 256** 버킷도 고려할 것

## 6. 버킷당 파일 수 — 카디널리티 반영 기준

### 6-1. 저카디널리티 파티션 (카디널리티 ≤ 10, 87% 데이터)

버킷 수를 아무리 높여도 **파일 수는 카디널리티에 수렴** 합니다.

| 파티션 데이터량 | 카디널리티 | 자연 생성 파일 수 | 파일 크기 | 조치 |
|---|---|---|---|---|
| 35 GB | 10 | ~10개 | 3.5 GB | ❌ Compaction 필수 |
| 15 GB | 10 | ~10개 | 1.5 GB | ❌ Compaction 필수 |
| 3 GB | 10 | ~10개 | 300 MB | ✅ 허용 |
| 1 GB | 5 | ~5개 | 200 MB | ✅ 허용 |
| 300 MB | 3 | ~3개 | 100 MB | ⚠️ Compaction 권장 |

**조치 기준**

```
파티션 데이터 > 5 GB이고 카디널리티 ≤ 10
→ 버킷 파일이 500 MB 초과 가능
→ Write 후 Compaction 으로 target-file-size 256MB 로 분할
```

### 6-2. 고카디널리티 파티션 (카디널리티 11 ~ 277, 13% 데이터)

bucket(64) 기준 실효 버킷 수 = 64

| 파티션 데이터량 | 버킷 64 파일 크기 | 버킷당 파일 수 목표 |
|---|---|---|
| 35 GB (코드A) | 547 MB/파일 | 1개 (약간 초과지만 허용) |
| 25 GB (코드B) | 390 MB/파일 | 1개 ✅ |
| 15 GB (코드C) | 234 MB/파일 | 1개 ✅ |
| 5 GB | 78 MB/파일 | ⚠️ Compaction으로 3~4개 → 1개로 병합 |

**버킷당 파일 수 권장 기준 (조회 성능)**

| 버킷당 파일 수 | 상태 | 조회 성능 영향 |
|---|---|---|
| **1개** | ✅✅ 최적 | Bucket Join shuffle 제거, 최소 file open |
| **2 ~ 3개** | ✅ 양호 | Pruning 유효, 파일 크기 ≥ 128 MB 이면 허용 |
| **4 ~ 5개** | ⚠️ 주의 | Join 최적화 반감, file open 비용 증가 |
| **6개 이상** | ❌ 비권장 | Manifest 증가, planning latency 상승, Compaction 필요 |

## 7. 데이터 구간별 통합 전략

### 구조 요약

```
전체 데이터 (1TB/day, 15 파티션)
│
├── 87% 데이터 — 카디널리티 ≤ 10 (소규모 파티션 다수)
│   └── 버킷 수 효과 없음 → 실효 파일 수 = 카디널리티 (~10개 이하)
│       ├── 파티션 데이터 < 3 GB  → 파일 크기 허용, 별도 조치 불필요
│       └── 파티션 데이터 ≥ 5 GB  → Write 후 Compaction 필수
│
└── 13% 데이터 — 카디널리티 11 ~ 277 (대용량 파티션)
    └── bucket(64) 효과 있음 → 실효 파일 수 = 64개
        ├── 코드A (35GB) : 547 MB/파일 → 허용 (또는 Compaction으로 2개 분할)
        ├── 코드B (25GB) : 390 MB/파일 → 권장 범위 ✅
        └── 코드C (15GB) : 234 MB/파일 → 권장 범위 ✅
```

### 결정 플로우차트

```
파티션의 버킷 컬럼 카디널리티 확인
│
├── 카디널리티 ≤ 설정 버킷 수 (87% 케이스)
│   → 버킷 수 변경 효과 없음
│   → 파티션 데이터량 확인
│       ├── 데이터 < 3 GB  → 현 상태 유지 ✅
│       └── 데이터 ≥ 5 GB  → 정기 Compaction 스케줄 설정
│
└── 카디널리티 > 설정 버킷 수 (13% 케이스)
    → 버킷 수 조정으로 파일 크기 제어 가능
    → bucket(64) 유지, 필요 시 128로 evolution 검토
```

## 8. 카디널리티별 Compaction 전략

### 저카디널리티 파티션 (카디널리티 ≤ 10, 대용량)

```sql
-- 파티션 데이터 5GB 초과, 카디널리티 ≤ 10인 경우
CALL system.rewrite_data_files(
  table => 'my_catalog.db.my_table',
  strategy => 'binpack',
  options => map(
    'target-file-size-bytes', '268435456',    -- 256 MB
    'min-file-size-bytes',    '134217728',    -- 128 MB
    'max-concurrent-file-group-rewrites', '5'
  ),
  where => "dt = '2026-01-01' AND 특정코드 IN ('저카디널리티_코드들')"
);
```


## 9. 카디널리티 모니터링 쿼리

### 파티션별 버킷 컬럼 카디널리티 확인

```python
from pyspark.sql import functions as F

df = spark.table("my_catalog.db.my_table").filter("dt = '2026-01-01'")

df.groupBy("특정코드") \
  .agg(
      F.countDistinct("버킷컬럼").alias("cardinality"),
      F.count("*").alias("row_count"),
      F.round(F.count("*") / 1e6, 1).alias("rows_million")
  ) \
  .withColumn("effective_buckets_64",  F.least(F.lit(64),  F.col("cardinality"))) \
  .withColumn("effective_buckets_32",  F.least(F.lit(32),  F.col("cardinality"))) \
  .withColumn("effective_buckets_16",  F.least(F.lit(16),  F.col("cardinality"))) \
  .orderBy(F.desc("row_count")) \
  .show(15, truncate=False)
```

### 파티션별 실제 파일 크기 확인

```sql
SELECT
  partition,
  count(*)                                          AS file_count,
  round(avg(file_size_in_bytes) / 1024 / 1024, 1)  AS avg_size_mb,
  round(max(file_size_in_bytes) / 1024 / 1024, 1)  AS max_size_mb,
  round(sum(file_size_in_bytes) / 1024 / 1024 / 1024, 2) AS total_gb
FROM my_catalog.db.my_table.files
WHERE partition.dt = '2026-01-01'
GROUP BY partition
ORDER BY total_gb DESC;
```

### 카디널리티 구간 분포 확인

```python
df.groupBy("특정코드") \
  .agg(F.countDistinct("버킷컬럼").alias("cardinality")) \
  .withColumn(
      "구간",
      F.when(F.col("cardinality") <= 10,  "≤10  (저카디널리티)")
       .when(F.col("cardinality") <= 50,  "11~50")
       .when(F.col("cardinality") <= 100, "51~100")
       .otherwise("101~277 (고카디널리티)")
  ) \
  .groupBy("구간") \
  .count() \
  .orderBy("구간") \
  .show()
```

## 10. 최종 권장 요약

### 버킷 수 결론

| 버킷 수 | 판정 | 이유 |
|---|---|---|
| **16** | ❌ 배제 | 고카디널리티 파티션 파일 2.2 GB |
| **32** | ❌ 배제 | 고카디널리티 파티션 파일 1.1 GB |
| **64** | ✅ 채택 | 고카디널리티 파티션 파일 547 MB (허용), 저카디널리티는 Compaction 병행 |
| *128 (미테스트)* | 검토 권장 | 고카디널리티 파티션 파일 273 MB ✅, 저카디널리티엔 영향 없음 |

### 운영 정책

| 파티션 유형 | 카디널리티 | 데이터량 | 조치 |
|---|---|---|---|
| 고카디널리티 대용량 | 11 ~ 277 | ≥ 15 GB | bucket(64) 유지, 파일 크기 모니터링 |
| 저카디널리티 대용량 | ≤ 10 | ≥ 5 GB | bucket(64) + **Compaction 필수** |
| 저카디널리티 소용량 | ≤ 10 | < 3 GB | bucket(64), 현 상태 허용 |

### 핵심 인사이트

```
카디널리티 ≤ 10 이면 버킷 수는 파일 수를 제어하지 못합니다.
→ 이 구간에서는 Compaction이 버킷보다 더 중요한 파일 관리 수단

카디널리티 > 설정 버킷 수 이면 버킷이 효과를 발휘합니다.
→ 고카디널리티(13%) 파티션에서 bucket(64)는 유효
```
